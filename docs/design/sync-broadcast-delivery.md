# Design: Intelligence → Sync Broadcast Delivery

**Status:** Proposed (implementation-ready)
**Phase:** 13 (WebSocket Architecture) — the "Intelligence → Sync message delivery" item
**Depends on / pairs with:** Phase 11 decision-state single-owner move (Sync as system of record)
**Author note:** Derived from a read of the current working tree on 2026-06-12. File/function references are verified against that tree.

---

## 1. Goal

Cards and agent messages produced by the Intelligence service must reach the user's connected client(s) through the Sync WebSocket hub — not through Intelligence's own (deprecated) `/chat/ws` endpoint. Per the design decision log, **Sync owns the client-facing WebSocket**; Intelligence is stateless relative to client connections and delivers to clients by publishing a NATS event that Sync consumes and fans out.

This document specifies that delivery path end-to-end and is meant to be implemented in one focused pass with a working Go + Python toolchain.

## 2. Current seams (what exists today)

| Component | File | State |
|-----------|------|-------|
| Sync NATS consumer | `sync/internal/nats/consumer.go` | Has a per-subject handler registry (`RegisterHandler`). Already registers `intelligence.card.created → handleCardCreated` and `email.sent → handleEmailSent`. |
| `handleCardCreated` | `sync/internal/nats/consumer.go` | **Stub** — unmarshals a compression-shaped card and logs/acks. Has **no hub reference**, so it cannot reach WS clients. |
| Sync hub | `sync/internal/websocket/hub.go` | `BroadcastToUser(userID, *models.WSEvent)` and `SendToDevice(...)` exist. Local fan-out via `broadcast` channel + cross-instance via Redis (`WSPublish`). Low-level `client.send` is a `chan []byte`. |
| `models.WSEvent` | `sync/internal/models/models.go` | Narrow struct: `Type, CardID, Text, TriggerWord, CursorPosition, Timestamp`. **Cannot carry a card/message payload.** |
| Consumer construction | `sync/cmd/server/main.go` | `natsConsumer` is created **before** the hub and is never given a hub reference. |
| Intelligence card publish | `intelligence/app/compression/service.py` | Publishes a `CreateCard` event to subject **`cards.created`** via `self.nats.publish(...)`. |
| Intelligence NATS client | `intelligence/infra/queue/nats_client.py` | Async `publish(subject, payload)`. |
| Client receiver | `client/src/hooks/useWebSocket.ts` | Expects messages shaped `{type, message:{id, sender_type, message_type, content_text, card_payload, created_at}, session_id, card_resolved}`. Now connects to Sync `/ws` (Phase 13 client fix, done). |

### Three mismatches to reconcile

1. **Subject mismatch.** Intelligence publishes `cards.created`; Sync registers `intelligence.card.created`. They never meet.
2. **Payload-shape mismatch.** The hub broadcasts a typed `WSEvent` that has no field for the client's `message`/`card_payload`. The client cannot render a `WSEvent`.
3. **Wiring gap.** The consumer has no hub reference, so even a correct handler cannot fan out.

## 3. Design decisions

### D1 — Deliver **by user**, not by session
Background cards are generated by the compression pipeline when there may be **no active WS session** (the user can be offline). Therefore delivery targets the **user**: `BroadcastToUser` already no-ops gracefully when the user has no connected devices, and Redis pub/sub covers cross-instance fan-out. `session_id`, when present, is passed through in the payload for client-side routing but is **not** a delivery filter. The user picks up queued cards from the stack when they next open a session.

> Implication: we do not need per-session connection lookup for this path. Keep it simple — fan out to all of a user's devices.

### D2 — Add a **raw-JSON broadcast path** to the hub (do not overload `WSEvent`)
`models.WSEvent` is purpose-built for paragraph-spawn streaming. Rather than bloat it with a polymorphic payload, add a parallel raw path:

- Extend the internal `broadcastMsg` with an optional `raw []byte`.
- In `routeBroadcast`, if `msg.raw != nil`, send `msg.raw` to `client.send` directly (skip `json.Marshal(msg.event)`).
- Add `func (h *Hub) BroadcastRawToUser(userID uuid.UUID, payload []byte) error` mirroring `BroadcastToUser` (local channel + `redis.WSPublish`).

This keeps the typed streaming API intact while letting the consumer forward an already-client-shaped JSON message verbatim. (Alternative considered: add `Payload json.RawMessage` + a `card` type to `WSEvent`. Rejected — it muddies a single-purpose struct and forces every consumer of `WSEvent` to handle a now-optional payload.)

### D3 — Standard envelope on a single subject: `sync.broadcast`
Intelligence publishes to the subject **`sync.broadcast`** with a stable envelope:

```json
{
  "user_id": "<uuid>",
  "session_id": "<uuid|null>",
  "event": {
    "type": "card",
    "session_id": "<uuid|null>",
    "message": {
      "id": "<uuid>",
      "sender_type": "agent",
      "message_type": "card",
      "content_text": "",
      "card_payload": { /* to_message_payload() / compression card shape */ },
      "created_at": "<rfc3339>"
    }
  }
}
```

- `user_id` is the delivery key (D1).
- `event` is **exactly** what the client receives (the consumer forwards it verbatim as the raw bytes in D2). This makes the contract one-hop and debuggable: what Intelligence emits under `event` is what `useWebSocket.ts` parses.
- Keep `intelligence.card.created`/`cards.created` for any *domain* side effects (persistence, metrics), but **delivery** rides `sync.broadcast`. (We could also have Sync derive `sync.broadcast` from `cards.created`; chosen against, because the compression card shape is not client-renderable without a reshape step, and we want the reshape to live in Intelligence next to the card generator, not in Sync.)

### D4 — Reshape lives in Intelligence
Both card-generation paths (`decision_stack/service.py` `to_message_payload` and `compression/service.py`) already know the client-facing payload best. They build the `event` envelope and publish it. Sync stays a dumb, fast fan-out. (This also means when Phase 16 unifies the two card paths, only Intelligence changes — Sync's delivery code is payload-agnostic.)

### D5 — Subject naming + stream
`sync.broadcast` must live on a JetStream stream the Sync consumer subscribes to. Verify the stream subject filter in `consumer.go` `ensureStream` (`Subjects: []string{stream + ".*>"}`) covers `sync.broadcast`, or add it. Durable consumer name follows the existing `sync-<sanitized-subject>` convention.

## 4. Message contract (authoritative)

The client (`useWebSocket.ts`) already branches on `data.type`. The `event` object in the envelope MUST use these types:

| `event.type` | Client effect | Required fields under `event` |
|--------------|---------------|-------------------------------|
| `card` / `message` | Appends an agent message (renders `card_payload` if present) | `message{ id, sender_type, message_type, content_text, card_payload, created_at }`, optional `session_id` |
| `card_action_result` | Appends agent message + resolves a card | `message{…}`, `card_resolved{ card_id }` |
| `typing` | Toggles typing indicator | `active: bool`, `session_id` |
| `source_email` | Attaches a source email to a message | `message_id`, `email` |
| `stack_complete` | System "stack complete" line | — |
| `error` | System error line | `message: string` |

No client change is required if Intelligence emits these shapes. (The hook is already updated for Sync connectivity.)

## 5. File-by-file implementation plan

### Sync (Go)

1. **`sync/internal/websocket/hub.go`**
   - Add `raw []byte` to the `broadcastMsg` struct.
   - `routeBroadcast`: `data := msg.raw; if data == nil { data, err = json.Marshal(msg.event) … }` then the existing device/user send loops are unchanged.
   - Add `BroadcastRawToUser(userID uuid.UUID, payload []byte) error` (local `broadcast` channel send + `redis.WSPublish(ctx, userID, payload)`).

2. **`sync/internal/nats/consumer.go`**
   - Give the consumer an optional broadcast sink: add a field `broadcast func(userID uuid.UUID, payload []byte) error` and a setter `SetBroadcaster(fn)` (avoids an import cycle with `websocket`).
   - Register `RegisterHandler("sync.broadcast", c.handleSyncBroadcast)`.
   - Implement `handleSyncBroadcast`: unmarshal `{ user_id, event }`, re-marshal `event` to bytes, call `c.broadcast(userID, eventBytes)` if set. Ack on success; treat malformed payloads as non-retryable (ack + log).
   - `ensureStream`/subject filter: confirm `sync.broadcast` is covered.

3. **`sync/cmd/server/main.go`**
   - After `wsHub` is created, wire it: `natsConsumer.SetBroadcaster(func(uid uuid.UUID, p []byte) error { return wsHub.BroadcastRawToUser(uid, p) })`.
   - Ordering: `NewConsumer` may stay where it is (before the hub) as long as `SetBroadcaster` is called **before** `natsConsumer.Subscribe(ctx)`. Today `Subscribe` is called after the hub is built — good; insert `SetBroadcaster` just before it.

### Intelligence (Python)

4. **`intelligence/app/compression/service.py`**
   - Alongside (or instead of) `cards.created`, publish a `sync.broadcast` envelope: build `event` with `type:"card"`, `message.card_payload` = the client-facing card payload, `created_at` = now. Publish via `self.nats.publish("sync.broadcast", envelope)`.
   - Reuse a single helper (see #6) so both card paths emit an identical envelope.

5. **`intelligence/app/agent/orchestrator.py`** (and/or `decision_stack/service.py`)
   - Where cards/messages are produced for a live session (`activate_next` → `to_message_payload`, send-approval results, etc.), also emit the same `sync.broadcast` envelope so they reach other devices of the same user. For request/response over the client's own socket this is optional, but it is what makes multi-device consistent.

6. **`intelligence/.../broadcast.py`** (new, small)
   - `def build_event(message_payload, *, type="card", session_id=None) -> dict` and `async def publish_broadcast(nats, user_id, event, session_id=None)`. One place that defines the envelope so Sync's contract has exactly one producer-side definition.

## 6. Edge cases & invariants

- **Offline user:** `BroadcastRawToUser` finds no devices → no-op locally; Redis publish still fires (harmless; other instances also no-op). Card remains queued in the stack and is delivered when the user opens a session. ✔ No special handling.
- **Cross-instance:** Redis `WSPublish(userID, payload)` already fans out; the raw path reuses it, so a user connected to instance B receives a card generated via instance A. Confirm the Redis subscriber on the receiving side forwards the raw bytes to `client.send` unchanged (it should, since payload is already client-shaped).
- **Single-connection-per-device:** unchanged; delivery iterates whatever devices are registered.
- **Backpressure:** existing `select { case client.send <- data: default: unregister }` applies to the raw path too — a slow client is dropped, not blocked.
- **Ordering:** JetStream per-subject ordering is sufficient; cards are independent, so strict global ordering is not required.
- **Security:** `user_id` in the envelope is trusted because it originates from Intelligence inside the mesh. Do **not** accept a client-supplied `user_id`; the WS connection's authenticated `userID` (from JWT) governs what that socket receives — the hub only sends to the matching user's connections, so a spoofed envelope could only target a user, never cross to an unauthorized socket.

## 7. Completion gate

1. `sync` `go build ./...` and `go test ./...` clean; `intelligence` `py_compile` clean.
2. Unit: a `sync.broadcast` message with a known `user_id` and `event` causes `handleSyncBroadcast` to call the broadcaster with the verbatim `event` bytes (table test with a fake broadcaster).
3. Unit: `routeBroadcast` with `raw != nil` delivers `raw` unchanged to a fake client `send` channel.
4. Integration (manual, one Gmail account): a card generated by the compression pipeline appears in the connected web client's stream **via Sync :8082** (Intelligence `/chat/ws` not involved).
5. Two clients authenticated as the same user both receive the card (validates D1 + Redis path).

---

## 8. Out of scope (explicitly deferred)

- Unifying the two card-generation models (`decision_stack` vs `compression`) — that is **Phase 16**. This design deliberately puts the envelope reshape behind a single helper (#6) so the unification touches only that helper.
- Decision-state single-owner move (Intelligence stateless, Sync system of record) — **Phase 11 remainder**. This delivery path is compatible with it: once Sync owns card state, `handleSyncBroadcast` can additionally persist before fan-out, but no redesign is needed.
- `notify` (urgent-interrupt) routing — separate UX, separate subject.
